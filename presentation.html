Why context is the hinge

“Where do my job outputs go?”

A batch job emits big outputs and small summaries. Where should each land?

A generic LLM

“Write them next to your job” → $HOME. The big files blow the home quota in minutes.

An LLM that knows the facility

Big outputs → fast local scratch (SSD/NVMe, wiped after the job); small summaries → the small backed-up home that persists.

The axes it reasons over: ephemeral vs persistent · local SSD/NVMe vs network-mounted (dCache, XCache) · user vs project space · small+quota'd vs bulk.

The point

The model didn’t get smarter — we gave it facility context. Hold this example; every later idea is a way to deliver context like this safely and portably.

Agent

A loop that uses an LLM to decide a step, call a tool, read the result, repeat.

MCP server

Model Context Protocol: a standard adapter exposing a service (Rucio, a notebook, a DB) as tools + live context any agent can call.

Agenda · the platform boundary

A credential-brokered MCP gateway

One endpoint in front of the facility’s services. The LLM client never holds raw credentials; the broker authenticates the user, mints per-service credentials, and audits every tool call.

client  ──►  mcp.<facility>  (oauth2-proxy → Keycloak OIDC)
            │
            ▼   FastMCP aggregator + credential broker
            ├── rucio-mcp     ├── condor-mcp
            ├── ami / metadata ├── jupyter-control
            └── … Nth backend (no code change)

This is the strategic boundary: identity, authZ, credential brokering, and audit in one place. It’s the natural way to put an agentic layer in front of a Tier-1 / PDP service estate without handing tokens to a chatbot.

Built on github.com/kratsg/af-mcp-platform (see research.md §AF-MCP). Real numbers: ~800 users, single endpoint mcp.af.uchicago.edu, brokers to Rucio/PanDA/AMI/GitLab/Jupyter/HTCondor. Agenda · identity is the bottleneck

Who is this agent, and may it act as you?

  • WLCG is mid-transition x509 → OIDC tokens; agents make identity the gating problem, not an afterthought.
  • How does a server trust a client it has never seen? Two answers: DCR (dynamic client registration — stateful, a /register endpoint, operational sprawl) vs CIMD (the client_id is an HTTPS URL to a metadata doc the server fetches — stateless, domain-trust).
  • I favour CIMD: no unbounded client DB, stable per-app identity, server-side allow-listing of trusted domains. It’s the new MCP default (still firming up; SSRF must be handled).

This is a place to shape the standard, not just consume it — a facility/Tier-1 voice in how research infrastructure adopts token and client-identity models.

NOTE: Giordon wrote "CIDM"; the spec term is CIMD (Client ID Metadata Documents), SEP-991, MCP spec 2025-11-25. Full detail + tradeoffs (SSRF, localhost, domain≠reputation, FastMCP cimd_trust_policy allowlist) in research.md §CIMD. "for obvious reasons" = a facility serving ~800 users + many clients does NOT want a stateful per-client registration DB; CIMD lets Nikhef allow-list trusted clients (claude.ai, cursor.com, internal apps) and keep audit clean. Deep-dive · CIMD vs DCR

Client ID = a URL the server fetches

GET /authorize?client_id=https://app.example/oauth.json&…
        │  authn server fetches the JSON, validates, applies policy
        ▼
{ "client_id":"https://app.example/oauth.json",
  "client_name":"…","redirect_uris":[…] }   ← hosted by the client
  • Wins: no /register abuse surface; stable identity per app; domain-based allow/deny.
  • Costs: server makes outbound fetches → SSRF risk (block private IPs, HTTPS-only, size/timeout caps); localhost ambiguity; domain ≠ trust.
BACKUP. Sources: MCP blog, Auth0/WorkOS/Stytch writeups, SEP-991, IETF draft (research.md §CIMD). Software statements / OS attestation are the next layer if asked.

Agenda · access without keys

ssh-less, key-less: log in with your federated identity

  • SSH keys are trusted permanently and spread across devices/teams — a poor fit for federated research infrastructure.
  • ssh-oidc turns an OIDC token into SSH access two ways: a PAM module, or short-lived SSH certificates from an online CA (oinit), with motley-cue mapping federated identity → local account and provisioning just-in-time.
  • Bonus: “non-username” login and it still works with rsync/git-over-ssh.

The push: OIDC tokens + SSH certificates across the toolchain, so the same identity that authorises a dataset read also authorises a shell — and an agent inherits it cleanly.

KIT/EOSC-synergy work (Marcus Hardt et al.); WLCG x509→token transition is the backdrop. Components: oidc-agent (client), motley-cue (server mapping/provisioning), pam-ssh-oidc, oinit (cert CA), mccli (wrapper). Indico talk Giordon flagged (indico.cern.ch/event/1471803/contributions/6967098) = the SSH-certificates push; cite it. Deep-dive · ssh-oidc internals

PAM token vs. SSH certificate

PAM (pam-ssh-oidc)

Paste/forward an access token; motley-cue verifies it; OTP fallback when token > 1023 B (OpenSSH limit). Unmodified sshd.

Certificate (oinit)

Online CA issues a short-lived SSH cert in exchange for a token; stored in ssh-agent; transparent for rsync/git; no per-host key trust.

BACKUP. Both rely on motley-cue for identity→account mapping + JIT provisioning. EPJ paper + KIT talks in research.md.

Agenda · what may an agent do?

Sandboxing on (and off) Kubernetes

On k8s

Agent tool-execution in throwaway pods: resource caps, network policy, no host access. The facility’s default isolation.

Off k8s

bubblewrap (the sandbox behind Flatpak) or sydbox: fence a process from files/network/hardware unless granted. Lightweight isolation on bare login nodes (cf. “blaude”, a bubblewrapped Claude).

The honest gap

Today a harness can advise (managed CLAUDE.md, allow-lists) but can’t reliably hard-deny a command at the user level. Real blocking needs a privileged policy engine on the node, or a sandbox. The instructions help; they are not enforcement — an agent just lets users 100× the bad behaviour we’ve had for decades.

Graded trust: human-in-the-loop gates for write/submit/email; privileged vs user-space lanes for ops agents.

Deep-dive · how do you actually enforce it?

Pods, cgroups, and eBPF

Coarse: containers / cgroups

Pods give resource isolation (CPU/mem/io via cgroups), bind-mounts to hide paths, seccomp to drop syscalls. Blunt but simple.

Fine: eBPF

Attach to tracepoints / LSM hooks; inspect — and now, via BPF-LSM, partly mutate — syscalls and file opens. Tetragon, Falco, bpflock. Powerful, complex.

cgroups/pods and eBPF are orthogonal: resource isolation vs fine-grained syscall/file policy. An agent may need both — a pod for blast-radius, a policy engine for “don’t find / the shared filesystem.”

BACKUP for the sysadmin/security crowd (Campana likely probes here). From Greg Thain (HTCondor): eBPF was historically read-only; BPF-LSM now allows some mutation of kernel state / syscall results (already replaced k8s networking proxy sidecars with in-kernel control; LSM hooks for fine-grained policy). Lincoln Bryant: cgroups = resource isolation, eBPF = attach to kernel funcs/data. Greg's HTC'24 eBPF talk is the reference. Tools: Tetragon, Falco, bpflock; sydbox (gitlab.exherbo.org/sydbox/sydbox); bubblewrap. research.md §sandboxing has full notes.

Agenda · portability of environments

pixi-native facility operations

  • The classic failure: “works on the login node, breaks on the worker.” Batch nodes have small memory, tight storage, and often no public internet.
  • pixi gives every project a locked, reproducible conda+PyPI environment; pixi-pack ships that exact environment into a batch job as a self-contained artifact (no CVMFS, no network needed).
  • Backed by conda-forge + the HEP Packaging Community: ARM/HPC builds, supply-chain hardening (dependency cooldowns).

Why it matters for agents: an agent that builds an analysis must ship the same environment it tested to the cluster. pixi makes “the env the agent used” a portable, verifiable object.

Agenda · a security reality

Agents don’t get the open web

  • Facilities (NERSC, UChicago, SLAC, BNL, AmSC, NRP…) are blocking web access for agents to defend against prompt injection from untrusted pages.
  • So a skill can’t just say “go read the docs.” It must encode just enough guidance — and delegate “what’s the latest?” to a curated MCP, not the open internet.
  • Not use /path/2024-egamma-recs.config (stale fast). Instead: “ask the chATLAS MCP for the latest recommendation path, then configure.”

The design rule: minimal durable guidance + a trusted live source. What an agent may reach is a facility decision, not a user convenience.

From the Giovanni Guerrieri / ATLAS-AMG email thread + Giordon's reply (Jun 2026). Core point: the "just point AI at the docs + basic guardrails" instinct fails because AFs block agent web access (prompt-injection hardening); hence skills hard-code "just enough" and use an MCP (e.g. chATLAS) for the latest paths/recommendations. Curated webfetch via an MCP (cf. Giovanni's cern-mkdocs-mcp docs_sources.json) is the controlled alternative to open web access. research.md §skills-governance. Agenda · where context lives

One commons, not a dozen forks

  • Skills are the durable context layer — but who hosts them, who maintains them, and how do we stop every experiment forking the same skill?
  • A live debate in ATLAS right now: a single-source marketplace (reference skills by URL, fix upstream) vs duplicated copies; docs kept next to the source code; official, e-group-governed repos over personal ones.
