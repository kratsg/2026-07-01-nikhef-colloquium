Why context is the hinge

“Where do my job outputs go?”

A batch job emits big outputs and small summaries. Where should each land?

A generic LLM

“Write them next to your job” → $HOME. The big files blow the home quota in minutes.

An LLM that knows the facility

Big outputs → fast local scratch (SSD/NVMe, wiped after the job); small summaries → the small backed-up home that persists.

Agenda · identity is the bottleneck

Who is this client, and may it act as you?

Every MCP client — Claude, an internal tool, something a student wrote — wants to connect. Once agents are the ones connecting, this stops being a one-time setup step and becomes a continuous, scalable decision:

Who is it?a durable identity, not just a name
Do we trust it?policy, not vibes
How much authority?scoped, auditable

Identity is becoming the bottleneck, not an afterthought — exactly the identity half of the Physics Data Processing problem, and WLCG’s x509→OIDC transition is the backdrop. Today, most clients register the way DCR prescribes — the adopted, stateful mechanism. A newer draft spec, CIMD, proposes something stateless instead (a client’s identity as a URL the server fetches) — worth watching, not yet the default. This sits on top of the community’s federated-identity work — WLCG token profiles, the AARC blueprint, IAM — not instead of it.

Deep-dive · CIMD vs DCR

Client ID = a URL the server fetches

GET /authorize?client_id=https://app.example/oauth.json&…
        │  authn server fetches the JSON, validates, applies policy
        ▼
{ "client_id":"https://app.example/oauth.json",
  "client_name":"…","redirect_uris":[…] }   ← hosted by the client
  • Wins: no /register abuse surface; stable identity per app; domain-based allow/deny.
  • Costs: server makes outbound fetches → SSRF risk (block private IPs, HTTPS-only, size/timeout caps); localhost ambiguity; domain ≠ trust.

Concretely in FastMCP: CIMDTrustPolicy(trusted_domains=["claude.ai","cursor.com"], auto_approve_trusted=True) — the facility decides which domains skip consent.

Auth today reuses grid creds (x509) or Keycloak OIDC — OAuth bridge docs.
BACKUP. The client_id IS the HTTPS URL to a metadata doc the client hosts; the authn server fetches it on demand, validates client_id == source URL and the redirect_uris, then applies its own policy. Read-only, stateless, fetch-on-demand. Spec status: adopted into MCP 2025-11-25 via SEP-991, from an IETF draft promoted to WG status Oct 2025 (origins in IndieAuth; in production at Bluesky); MCP Python SDK added minimal client-side support in v1.23.0, server-side still maturing. Stance: CIMD = SHOULD, DCR = MAY, pre-registration still allowed. Software statements (RFC 7591 §2.3) / OS attestation are the next layer if asked. Agenda · access without keys

ssh-less, key-less: log in with your federated identity

SSH keys are trusted permanently, live on disk or in an agent socket, and spread across devices and teams — and now there’s a new reader in the room: an AI agent with shell access. ssh-oidc (KIT/EOSC-synergy, AARC-native) is one toolkit that replaces the permanent key with a short-lived, revocable, federated credential.

Private keyon disk / in ssh-agent, forever
Anything with agent accessincl. an LLM agent
Can sign in as youno expiry, no federation, hard to revoke

Federated OIDC tokenminutes, from your IdP
motley-cueidentity → account, JIT
PAM token or SSH certpam-ssh-oidc / oinit — short-lived, revocable

The push: OIDC tokens + SSH certificates across the toolchain, so the same identity that authorises a dataset read also authorises a shell — and an agent inherits a short-lived credential, never a permanent private key.

ssh-oidc: Gudu, Hardt et al. (KIT / EOSC-synergy). “SSH with federated identities,” CHEP 2026; EPJ Res. Infra. 2025, doi:10.1007/s41781-025-00136-5.
KIT/EOSC-synergy work (Diana Gudu, Marcus Hardt et al.); WLCG x509→token transition is the backdrop. THE DIAGRAM'S POINT (make it explicit): pam-ssh-oidc, oinit, motley-cue, and oidc-agent are NOT four competing tools — they're all ONE toolkit (ssh-oidc). oidc-agent obtains/holds the federated token client-side; motley-cue is the server daemon that maps that identity to a local account and provisions it just-in-time; pam-ssh-oidc and oinit are the two alternative LAST STEPS (PAM challenge-response vs a short-lived SSH certificate from an online CA) that actually get you a shell. mccli wraps oidc-agent+ motley-cue together for the end user. THE AGENT-SAFETY ANGLE (new, say it explicitly): today, anything with access to your ssh-agent socket — including a coding agent you gave shell access to — can ask it to sign with your permanent private key. ssh-oidc's credential is short-lived and federated: even if an agent touches it, the blast radius is minutes, not forever, and it's revocable at the identity provider instead of requiring a key rotation across every host you've ever used. Deep-dive · ssh-oidc internals

PAM token vs. SSH certificate

Both are last-mile options within the same ssh-oidc toolkit — oidc-agent holds the token client-side, motley-cue maps identity→account server-side and provisions it just-in-time, then one of these two gets you a shell.

PAM (pam-ssh-oidc)

Paste/forward an access token; motley-cue verifies it; OTP fallback when token > 1023 B (OpenSSH limit). Unmodified sshd.

Certificate (oinit)

Online CA issues a short-lived SSH cert in exchange for a token; stored in ssh-agent; transparent for rsync/git; no per-host key trust; enables “non-username” login.

mccli wraps the whole toolkit into one command for the end user:

mccli ssh login.af.uchicago.edu
  → oidc-agent supplies the token, motley-cue maps you to an account, you get a shell.

Deep-dive · how do you actually enforce it?

Pods, cgroups, and eBPF

Coarse: containers / cgroups

Resource isolation (CPU/mem/io), bind-mounts to hide paths, seccomp to drop syscalls. Blunt but simple.

Fine: eBPF

Sandboxed kernel programs attached to a hook point — a syscall, a kernel tracepoint, a kprobe/uprobe almost anywhere. Tetragon, Falco, bpflock.

cgroups/pods and eBPF are orthogonal: resource isolation vs fine-grained syscall/file policy. An agent may need both — a pod for blast-radius, a policy engine for “don’t find / the shared filesystem.”

eBPF: ebpf.io/what-is-ebpf · agent-focused eBPF observability: agentsight.
BACKUP for the sysadmin/security crowd (Campana likely probes here). Diagram redrawn from ebpf.io's own hook-point explanation: "eBPF programs are event-driven and run when the kernel or an application passes a certain hook point. Pre-defined hooks include system calls, function entry/exit, kernel tracepoints, network events; if none exists, attach a kprobe/uprobe almost anywhere." From Greg Thain (HTCondor): eBPF was historically read-only; BPF-LSM now allows some mutation of kernel state/syscall results (already replaced k8s networking-proxy sidecars with in-kernel control). Lincoln Bryant: cgroups = resource isolation, eBPF = attach to kernel funcs/data. Greg's HTC'24 eBPF talk is the reference. Tools: Tetragon, Falco, bpflock; sydbox (gitlab.exherbo.org/sydbox/sydbox); bubblewrap. NEW: agentsight (eunomia-bpf) is a perf/top/strace-like tool FOR AI AGENTS specifically — eBPF probes watch an agent's process tree, file ops, network destinations, and (via TLS tracing at the syscall boundary) even the plaintext LLM request/response, all without modifying the agent.
